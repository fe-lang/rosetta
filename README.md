# rosetta-fe

Side-by-side Solidity ↔ Fe implementations of real-world Ethereum contracts, with Foundry-based gas benchmarks and differential equivalence testing.

Each module takes a production Solidity contract (or JS-generated bytecode), implements the same logic in [Fe](https://fe-lang.org), and proves they're equivalent — then measures gas.

## Why these contracts?

Solidity and Vyper were designed for tokens and early DeFi. As Ethereum's use cases expanded — ZK proofs, on-chain games, complex math, upgradability patterns — developers hit the language's limits and resorted to inline assembly, JS code generation, or raw bytecode.

Fe handles these cases natively. Each module here demonstrates a real pain point and how Fe resolves it.

See [Fe Use Cases](https://fe-lang.org) for the full rationale.

## Structure

Each module follows the same pattern:

```
module/
├── fe/                    # Fe implementation (ingot)
│   ├── fe.toml
│   └── src/
├── bench/                 # Foundry project
│   ├── foundry.toml
│   ├── src/               # Solidity reference implementation
│   └── test/              # Differential tests + gas benchmarks
└── README.md              # Gas report + what Fe does differently
```

### Running benchmarks

Each module's bench/ directory is a standalone Foundry project:

```bash
cd module/bench
FE_SONA_OPT_LEVEL=2 forge test --ffi --offline -vvv
```

Requirements:
- [Fe compiler](https://github.com/ethereum/fe) (`fe` on PATH)
- [Foundry](https://book.getfoundry.sh/) (`forge`)
- `solc` 0.8.24+

## What "equivalence" means

For every module, the Foundry test suite verifies:
- **Identical outputs** for identical inputs across Fe and Solidity implementations
- **Identical reverts** for invalid inputs
- **Differential fuzzing** (1024+ runs per function by default)
- **Gas measurement** using Foundry's `pauseGasMetering` / `resumeGasMetering`

This is strong empirical validation, not formal proof.

## Why Fe wins on gas

Fe's Sonatina backend compiles directly to EVM bytecode (no Solidity IR detour). Combined with monomorphization of traits and const generics, the compiler produces tight bytecode from clean, readable source. No assembly required.

## Contributing

Pick a contract from the [target list](./TARGETS.md), implement it in Fe, add a Foundry bench, and open a PR. The Fe implementation should:
- Match the Solidity ABI exactly (for differential testing)
- Use Fe idioms (traits, const generics, effects) where they improve the code
- Not sacrifice readability for gas micro-optimization
