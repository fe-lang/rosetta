# Design Notes

What idiomatic Fe code would look like for these modules, and what's needed to get there.

## Observations

### Type System Opportunities

Fe has const generics, traits with associated types, HKTs, effects, and `const fn`. The examples should show this off:

- **Const-generic field type**: `Fp<const P: u256>` would make the prime a type parameter, preventing accidental mixing of BN254 scalar field and base field arithmetic. Two `Fp<F_Q>` values can't be added to an `Fp<F_R>` at compile time. This is something Solidity literally cannot express.

- **Trait-based hash abstraction**: `trait Hasher { fn update(mut self, data: u256); fn finish(self) -> u256 }` with impls for SHA256 and Keccak. Merkle proofs and Fiat-Shamir transcripts become generic over the hash.

- **Const fn round constants**: Poseidon's round constants could be generated by `const fn` rather than hardcoded. The MDS matrix, S-box constants, etc. could be derived from the field parameters at compile time. This is a massive readability win.

- **Effects for precompile access**: EC operations could use Fe's effect system to express that `ec_add` requires the `Precompile` effect, making the dependency on EVM precompiles explicit in the type signature.

### Aesthetic / DX Observations

- `addmod(a, b, P)` and `mulmod(a, b, P)` passing `P` every time is noisy. An `Fp` type or even just `fp_add(a, b)` would be cleaner. (Blocked by sonatina#227.)

- Manual memory management in the transcript (`ops::mstore(ptr + offset, value)`) is the worst part of the Plonk verifier. A byte buffer type would eliminate ~40 lines of offset arithmetic.

- `ops::staticcall(ops::gas(), 2, ptr, len, out, 32)` for SHA256 is write-only code. `sha256(ptr, len)` is what it should be. (Added to `std::evm::crypto`.)

- The `hash_fr` function writes "BSB22-Plonk" byte-by-byte with 11 `mstore8` calls. A const byte array literal would be cleaner: `const DST: [u8; 11] = "BSB22-Plonk"`. Does Fe have string-to-byte-array literals?

- Const arrays of `u256` should ideally be `const C: [Fp; 81]` if `Fp` is `Copy` and `const fn`-constructible.

- The `own` keyword on `Add`/`Mul` trait methods feels heavy for `Copy` types. Could Fe infer that `own` is a no-op when `T: Copy`?

### What's Missing in std

1. **SHA256 precompile** — added to `std::evm::crypto` (this session)
2. **Byte buffer / writer** for building precompile inputs without manual mstore
3. **BN254 EC operations** — currently in rosetta-fe `ec` ingot, should be in std
4. **EC pairing** — the pairing check precompile (0x08) needs a wrapper
5. **Modular arithmetic** — `Fp<const P: u256>` with operator traits

### Compiler Blockers

1. **Stack object overflow** — sonatina panics on wrapper function composition at scale (filed on sonatina#227). Blocks `Fp` type, blocks idiomatic field arithmetic, blocks the most compelling Fe showcase.

2. **Seq::len() inlining** — loop bounds from `for x in 0..CONST` go through `Seq::len()` which isn't inlined, blocking loop unrolling and const-index folding.

3. **O2 miscompilation** — Plonk deployed contract returns wrong results at O1/O2.
